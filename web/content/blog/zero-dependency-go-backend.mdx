The Go API that powers this site has an unusual property: its `go.mod` lists zero
third-party dependencies. Not "few." None. Every line that runs — the rate limiter, the
SSE proxy to the model, config loading, the contact-form spam traps — is either mine or the
standard library's. That was a deliberate constraint, and it paid off in ways I didn't fully
expect.

## The rule

> No third-party Go modules. If the standard library can do it, the standard library does
> it. If it can't, I write the small thing I actually need.

It sounds austere. In practice the Go standard library is broad enough that the rule rarely
hurts. `net/http` is a real server, not a toy. `context` gives you deadlines and
cancellation. `crypto`, `encoding/json`, `sync`, `time` — the batteries are mostly included.

<Callout type="note">
  This is a small backend for a portfolio site, not a sprawling product. The rule scales
  down beautifully here. I'm not claiming you should run your payments platform this way.
</Callout>

## What it bought me

**The supply-chain surface is effectively zero.** There's no transitive dependency tree to
audit, no `npm audit`-equivalent treadmill, no 2 a.m. CVE in a logging library I forgot I
pulled in. The only code I have to trust is Go's and mine.

**I actually understand the moving parts.** When I wanted per-IP rate limiting, I didn't
import a middleware — I wrote windowed counters guarded by a mutex: one window per IP per
minute, another per IP per day, and a global daily cap. Because I wrote it, I know exactly
how it behaves under a burst, what it does on an IP it's never seen, and where the global
ceiling kicks in.

```go
// Per-IP + global limiting, standard library only — no middleware framework.
type Limiter struct {
    mu          sync.Mutex
    perIPMinute map[string]*window
    perIPDay    map[string]*window
    // …plus a global daily counter and a janitor that evicts stale windows.
}
```

**Upgrades are boring.** Bumping the Go version is the entire dependency update story.
There's no lockfile to reconcile, no breaking change in a package three levels deep.

## Where it gets interesting: the model proxy

The most "I'd normally reach for a library" moment was the LLM demo. The browser talks only
to my backend, which proxies an upstream OpenAI-style streaming endpoint and re-emits it to
the client as simplified Server-Sent Events. No SDK, no SSE library — just `net/http` and a
`bufio.Scanner` over the response body.

That turned out to be the *right* amount of code, not a heroic amount. It's also the only
place that parses the upstream chunk shape, so swapping models is a one-file change. (I went
deeper on the SSE-vs-WebSockets decision in
[another post](/blog/streaming-llm-tokens-over-sse).)

## The honest trade-off

The rule isn't free. I've written small versions of things a library would hand me —
frontmatter parsing on the web side, the rate limiter, the spam checks. If any of those grew
genuinely complex, the calculus would flip and I'd take the dependency. The discipline isn't
"never depend on anything"; it's **"don't import what you don't understand or can't replace
in an afternoon."** For a backend this size, that line sits at zero.

The result is a codebase I can hold entirely in my head — which, for the thing that runs my
own website, is exactly what I want.
